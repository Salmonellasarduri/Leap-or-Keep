# 立体ウォーテーブル化 — 盤を部屋の3D世界に実在させる(設計・実現可能性)

> 2026-07-14 策定。オーナー構想「盤を背景の前に置くのでなく、部屋の3D世界の中に実在させる=卓上のホロ・ウォーテーブルで、駒(実在する船)が3Dで動く。船外はリアル戦闘なので、慣性で船内が揺れ、被弾で赤アラート+衝撃が走る。船の艦橋から実戦を指揮している感覚」。
> Sol(GPT-5.6)がコンセプト、Claudeがコード地図+実現可能性(13エージェントのワークフロー)を担当。**結論=現エンジンで書き直し無しに実装可能。ブロッカー無し**。

## 感情の核(Sol)
**"embodied command"** — チェスの明晰さ+「自分もその駒の一つに乗っている」身体的な緊張。命令が重いのは、推力・衝撃・危険が「シミュレーション⇔船内」の境界を越えて伝わるから。

## なぜ実装可能か(コード地図の要点)
- ゲームの命令入口 `H.cellClick(x,y,dir)` / `H.unitClick(id)` は**整数座標のみ**でカメラ/ピクセルに一切依存しない(index.html:4718)。→ これは**描画+入力取得の変更であって、ゲームの書き直しではない**。
- 融合したい2つ(DOM固定の盤・一人称の部屋)は**既に両方 holo.js 内に存在**。`boardGroup`(holo.js:54)はカメラ非依存の自己完結サブツリー。
- 反応レイヤーのフック(`fx()` boom/flash/hitflash/shake holo.js:496、`disturb()` :509、`driftBeat`慣性フラグ :1043、段階shake `n=min(3,link)` index.html:1166、`room.lights`/`FogExp2`、`roomCam`)は**既に戦闘に配線済み**。

## 唯一の本当に難しい点(手法非依存)
**斜め視の可読性・押しやすさ**。盤を卓に寝かせて「卓上」に見せた瞬間、遠い行が縮む(390pxモバイルで奥3行が約10–12px=44pxタッチ目標を大きく下回る)+ラベルが斜めにせん断する。**これは matrix3d でも raycast でも同じ**(スクリーン幾何が同一だから raycast でも救えない)。→ 勝敗は**アートディレクション規律(どこまで寝かせるか+卓を大きく取るか)**で決まる=Claudeの管制領域。処方: 実効チルトを現行34°付近に留める/ウォーテーブルを画面下60%を占める大きさにする/卓は盤の額縁として扱う。

## 推奨アプローチ: 段階ハイブリッド(3審査員が全員1位)
DOMホモグラフィ(Approach A)を耐久的な終点にし、フル3D merge(Approach B)は任意の最終段として保持。理由: クリック精度は**幾何でなく組合せ**問題 — セルは `onclick="H.cellClick(x,y,dir)"` を焼き込み済みで、ブラウザは祖先のCSS変換越しにヒットテストするだけ(今の `perspective:1050px+rotateX(34deg)` と同じ原理)。卓に寝かせた盤(四隅がCSSカメラ前=w>0)なら **matrix3d でクリック精度は構造的に正しい**(アプリ側の逆投影ゼロ)。Aは4つの高コスト資産(DOMヒットテスト/ツールチップ/i18n MutationObserver/WebGL不可の2Dフォールバック)を温存。Bの唯一の勝ち=真の遮蔽(卓上の物が駒を隠す)+完全一致遠近だが、4系統の再構築+fps退行+結局WebGL不可用にDOMヒット層が要る。

## 段階プラン
- **Phase 1 — 卓に着地(ソケット)/ 労力S**: `bg.scene` にウォーテーブル台座+放射状の接地影を desk-top(three.js約(0,0.7375,-0.38)、Box3で実測)に置き、既存の「盤over部屋」合成を物理的に座って見せる。入力・ラベル・calibrate 一切不変。最小の可視的勝利。リスク: z-fight(数mm持ち上げ+polygonOffset)、2カメラ近似一致(目視調整)。
- **Phase 2 — 反応する船内(慣性揺れ+被弾赤アラート)/ 労力M**: 既存 `fx()` を部屋へfan-out。boom/hitflash→`room.lights`赤パルス+`FogExp2`スパイク、`roomCam` に減衰カメラシェイク項(段階shake+driftBeat慣性)。**Solの処方=ランダム揺れでなく方向性プッシュ**(前進推力→視点後退/横移動→横揺れ、0.15–0.35°/180–350ms)、被弾=鋭い方向キック0.5–1.2°+赤の非常灯1.5–2Hz+赤エッジ・ヴィネット(全面赤にしない)+ホロのせん断。reduced-motionトグル必須。**リスク=このプランで唯一の実リスク**: 光/フォグ/カメラ変更が `room.drawn` を無効化しビート中に重いPBR部屋を再描画→dirty-renderスキップ(holo.js:806-812)を破りモバイルfpsゲート(p95≤17.5ms)を脅かす。緩和: shakeは盤canvas側(CSS shake=部屋コスト0)、部屋再描画は赤アラートのみ~30fpsに絞る、`META.fxLite`ゲート、prefers-reduced-motion自動ON。**Phase 2後に `tools/fps.mjs` を再走させ p95≤17.5/hitches≤4 を必須ゲート**。
- **Phase 3 — 卓ロック・ホモグラフィ(Approach A: 駒がワールド空間・遠近一致・DOMクリック温存)/ 労力L**: 盤のCSS投影を毎フレーム `matrix3d`(`body.holo3` スコープ=WebGL不可時は従来フラット盤へ退避)に置換し、透明 `.cell` グリッドを desk quad に写像。`calibrate()` の off-axis frustum を「この homography を再現」へ一般化、`calibCheck`(≤2.5px)は視覚オラクルとして維持。**強制的な作り直し=ラベル/dirpick を別の非変換スクリーン空間DOMオーバレイへ持ち上げ**、各駒の投影点で毎フレーム配置(直立・指サイズ・DOMのまま=i18nとAT到達を維持)。
- **Phase 4 — 任意: フルraycast merge(Approach B)/ 労力XL**: 真の遮蔽が必須になった時だけ。`boardGroup` を `bg.scene` へ、`tileMesh` の `instanceId==y*GRID+x`(holo.js:190)で raycast→(x,y)、同じ `H.cellClick`。dirty-skip無効化/トーン再著述/DOMフォールバック層維持が代償。

## トップリスクと緩和
1. **斜め視でモバイルタッチが死ぬ(最難・手法非依存)** → チルトを34°付近に抑制+卓を大きく。Phase3を「390pxで各投影セル最小辺≥32–40px」測定スクリプトでゲート。深い寝かせが要る時はタップ曖昧化ポップアップ(dirpickパターン再利用)。
2. **反応fx/Ph4でfpsゲート退行** → 盤は常時描画canvasに残す/赤アラート再描画を~30fps・fxLiteゲート/各段後に fps.mjs 再走。動く盤を bg.scene へ入れない(Ph4を除く)。
3. **homographyでラベル/dirpickせん断** → Ph3の非変換再投影オーバレイ。
4. **preserve-3d z順で dirpick が押せない**(過去の既知バグ index.html:141-142「ラム方向選択が押せない」再燃) → 同オーバレイ+translateZ前進パターン再適用。誤ヒット=トーラス2way慣性の**誤操作**(安全な空振りでない)ので要注意。
5. **2カメラは完全一致しない** → 準トップダウンなら差は微小、接地ソケットで継ぎ目を隠す。完全一致はPh4のみ=遮蔽必須でなければ非推奨。
6. **不変条件は壊れない**: `calibCheck` は手動オラクル(CI非配線=CIは lint+sim+agent のみ)。クリック経路が組合せのまま=sim/agent/lint(L-037)不変。2Dフォールバックは matrix3d を `body.holo3` スコープに閉じDOM盤を描き続ける限り生存。

## 最初にやるべき最小実験(着手前の de-risk)
**静的な手調整 matrix3d を `#board` に(`body.holo3`スコープ)+ 390pxモバイルビューポートで開く ~20行スクリプト**: (a) 25セルの画面重心へ合成クリックを撃ち、各々が意図の `H.cellClick(x,y,dir)` に届くか検証(w>0なら通るはず=「逆投影ゼロで正しい」を実証)、(b) 各セルの `getBoundingClientRect` で行ごとの最小投影スクリーン寸法を報告。CSS文字列1本+計測ループのみ(holo.js/カメラ/raycast変更ゼロ)。**(b)が本命** — 最難の「斜め視タッチ目標」を**数値**で出し、Phase3着手前に安全なチルト/スケールを決められる。raycastも同じスクリーン幾何なので、この1実験で両終点の入力精度が同時に片付く。

## 最小実験の結果(2026-07-14 実測 — 想定を1つ覆した)
`tmp/warroom-probe.mjs`(390px級モバイル・holo盤のtilt掃引・DOM hit-test+セル寸法計測、holo.js無変更):

| 盤の寝かせ | クリック経路 | 奥行き行のタッチ目標 |
|---|---|---|
| **34°(現行・出荷中)** | **25/25 ✓** | 40px(可) |
| 45° | 23/25 ✗ | 32px |
| 55° | 14/25 ✗✗ | 24px |
| 65° | (20/25*) | 15px |

**重要な発見**: ワークフローは「w>0ならクリックは構造的に常に正しい」と推論したが、**実測は約40°を超えると崩れる**。盤を寝かせると奥のセルがスクリーン空間で前のセルと重なり、hit-testが別セル(や隙間)を返す=誤ヒット。**これはスクリーン幾何の問題なので three.js raycast でも同じ前セルに当たる**=raycast(Ph4)でも救えない。プロトタイプでしか出ない類の知見。

**設計への含意(本命の成果)**:
- **盤を卓に「寝かせない」**。現行34°が安全な縁。フラット化はクリック破綻+タッチ目標割れの袋小路。
- ウォーテーブルは「投影機ベースの上に**読める角度で立つホロ表示**(実在の傾いた戦術コンソール)」として卓に**座らせる**=見下ろす平盤にしない。むしろ本物の指揮卓に近く、可遊性を保つ。
- 故に高価値・低risk=**Ph1(投影機ソケットに着地)+Ph2(反応する船内)**はこの制約と無関係で不変。Ph3は**控えめな傾き**で遠近一致。Ph4(raycast)は真の遮蔽が要る時だけ(「寝かせ」には無力)。

## Phase 1 実装完了(2026-07-14 — Sol独立監査8/10「Phase1出荷可」)
`holo.js buildRoomRig()` = 卓上のサルベージ投影機ソケットを静的リグとして `bg.scene` に実在させ、盤ホロが卓の実機から立ち上がって見せる。
- **決定的発見**: 盤ホロ(前面canvas #holo-canvas)は部屋(背面canvas #holo-bg)の上に合成される→ソケットを盤直下に置くと**盤に遮蔽される**。可視化できるのは盤シルエットの外(前縁の下・下角の外)のみ。加えて盤はHUD次第で左右にズレる(卓の3D中心とは一致しない)。
- **解法**: `positionRig()` が盤DOM矩形の前端中央を部屋カメラで卓平面(y=DESK_Y=0.7375)へ**逆投影**し、リグを毎レイアウト盤直下へ追従(デスクトップ左寄せ/モバイル中央の両方で整合)。**矩形が変わった時のみ再配置=再描画**→dirty-renderスキップ維持=ビート中fpsコスト0。
- **造形(Sol処方2巡)**: 低広プリンス+露出**前面ファサード**(継ぎ目/大ボルト4/**アンバー計器灯**/彫込みシアンリップ)+**機械式エミッタ・ブラケット**(暗housing+足+クランプ+内向きシアンアパーチャ=発光はコアのみ・白飛び回避)+シアン投影スピル+広く柔い青寄り光の井戸+非発光エッジキャッチ(暗合成でのシルエット保持)。
- **ゲート**: モバイルfps p95 16.8ms/hitches 3(≤17.5/≤4)PASS、`npm test` 360 PASS、lint L-037 OK。入力/calibrate/2Dフォールバック不変。
- 検証ハーネス(gitignore): `tmp/phase1-shot.mjs`(合成)/`tmp/phase1-debug.mjs`(盤ホロ隠して部屋のみ)/`tmp/proj-calc.mjs`(投影計算)。デバッグ: `window.__holo.rigInfo()`。

## Solのステージング(ビジュアル)
卓上に**サルベージ製ホロ投影機ユニット**(45–60cm・不揃い装甲・露出ボルト・溶接補修・4本のシアン発光ポスト)、グリッドは発光面の2–4cm上・船は更に4–8cm上に浮遊、浅い**切頭光錐**(半透明シアン+ノイズ+走査帯+塵)。駒は**ミニチュア船シルエット**(識別可能な船体+明るいエンジン/艦橋/武装、被弾でホロ不安定化・ピクセル片剥離)。移動=浮上→目的地へバンク→着地→ホロ再走査+軌跡リボン。攻撃=抑制されたベクトル線+着弾ブルーム。右のHUDは卓横の**直立補助ディスプレイ**として世界内化=卓全体を1つの指揮ステーションに。

# Lessons — Leap or Keep

> 実プレイQA・開発で得た教訓。再発したら「シムの不変条件」へ昇格させる。形式: L-XXX / タグ / 教訓 / 対策。
>
> **卒業=機械化の原則**(隣プロジェクト `Artificial Personality/.claude/skills/lesson-record` の仕組みを本作に移植: `.claude/skills/lesson-record/SKILL.md`)。
> 散文の教訓は「従われない」。静的に検知できる罠はガードに卒業させ、毎コミット自動実行する:
> - `node tools/lint-guards.mjs` … LOGIC区間の純度(L-037)/ EN_RULES.unshift強制(L-038)を静的検査
> - `node tools/fps.mjs` … 負荷由来の偽FAILを自動再計測で切り分け(L-041)
> - `npm test` / `npm run test:agent` … 仕様不変条件(L-020/L-042 ほか)
> - `window.__holo.calibCheck()` … 盤の投影整合 ≤2.5px(L-039)
> 各レッスンの末尾に **卒業先** を書く。卒業先を持つレッスンは「再発は機械が止める」状態。

- **L-001** [rules][torus] 4×4トーラスでは「左に2」と「右に2」の着地マスが同一になり、マスクリック式の移動UIでは**ドリフト方向がプレイヤーの意図と食い違う**(find()が先に見つけた方向を採用していた)。→ 対策: 移動は方向明示(曖昧マスには方向矢印ボタン)。シム性質テスト「実行後のドリフト=指定方向」で恒久ガード。オーナー報告 2026-06-11「ドリフトの方向が正しくない」の根本原因。
- **L-002** [rules][gh] 「移動2で1マスだけ動く」(GHの up to N)は本作では**意図の読みにくさ+慣性の軽さ**につながる。オーナー裁定: 移動Nはちょうど N。→ moveOptions を exact-N に変更。「精密は有料」のピラーが強化される副作用も良。
- **L-003** [ux][telegraph] 攻撃予告(赤マス)は**回避が成立した瞬間のフィードバックが無いと「意味がない」と感じられる**(ItBは攻撃が空振るアニメで「避けた!」を作る)。→ 敵攻撃は赤マスを必ず光らせ、空振り時は「回避!」フロート表示。さらに性質テスト「赤マス外のプレイヤーは攻撃ダメージを受けない」で正直性を機械的に保証。
- **L-004** [ux][drift] ドリフト解決をpump内で瞬時に終わらせると**何が起きたか見えない**。ラウンド頭に専用ビート(アニメ+衝突エフェクト)を挟む。「ルールはアニメで教える」(ItB開発者)。
- **L-005** [tooling] **解決済(2026-06-11)**: Claude Preview の screenshot 不安定問題は、**自前のPlaywrightハーネス**(`node tools/shot.mjs <scenario>`)で恒久回避。同じページをPlaywrightは即座に撮れた=不調はMCPツール側と確定。教訓: **自走に必須のツールが不調なら、修理を待たず自前の代替経路を持つ**(ヘッドレスブラウザ+シナリオスクリプト+PNG出力→Readで視認)。シナリオは tools/shot.mjs の SCENARIOS に追加するだけ。
- **L-006** [ux] クリック数は1ターン≦6を基準にする(SPEC §10.2)。「次に何をクリックすべきか」が常に1箇所だけ光る状態を保つ。自動進行+デフォルト続行+Enter確定。
- **L-007** [qa] 「赤マス外で攻撃された」というプレイ報告は、性質テスト(600ラン)で**ロジック違反ゼロ**と判明 — 体感バグの正体は「回避成功時の無反応」だった。**プレイヤーの体感報告はまず機械検証で事実と知覚に切り分ける**。事実ならロジック修正、知覚ならフィードバック追加。
- **L-008** [balance] 移動を「きっかりN」にしたら deep(全展開潜行)の勝率が39%→24%に低下、greedy は帰還ゾーンが2→3に深化。移動ルールはバランス全体に波及する一級変数。変更時は必ず3方策シムで前後比較する。
- **L-009** [design] 「悩むところがない」の正体は大抵 ①上位互換の二択(使い得) ②非線形が存在しても**決断の瞬間に見えない** ③システム間の未接続(多脚不足)。修正は新ルール追加より「コストの追加・可視化・接続」が先(game-design-essence 監査チェックリストを回す)。展開遺物の喪失リスク化/アンロックラダー表示/クラッシュサルベージは全てこのパターン。
- **L-010** [process] 状態機械にフェーズ割込み(crashsalvage等)を足すときは、**復帰先のstepがphaseを再設定するか**を必ず確認(pump の case "player" が phase を据え置いてモーダル残留)。新フェーズ追加時の regression チェック項目に昇格。
- **L-011** [rules][torus] **偶数トーラス×偶数移動は方向が縮退する**(4×4で左2=右2)。方向選択UIで対処するより、奇数盤(5×5)にすれば構造的に消える。パラメータの組が生む縮退ケースは仕様時に列挙する(essence監査#10)。オーナー指摘「方向選べるの微妙/4×4狭い」の根本解。
- **L-012** [ux] **同じ種類の選択は常に同じ見た目のUIで**。カードを選ばせる場面(燃料・休息・燃やし・回収)は全てカード面表示に統一、移動系(通常移動・突進)は全てマス選択に統一。「この場面だけ別UI」はその都度学習コストになる。
- **L-013** [ux] プレイヤーが繰り返す入力(ロードアウト、デッキ編成)は**前回値をデフォルト**にする。毎回ゼロから選ばせるのは設計の手抜き。
- **L-014** [ux] **標準操作の約束事は最初から実装する**: 右クリック=1段階キャンセル、同じ対象の再クリック=選択解除、Esc=戻る、Enter=確定。プレイヤーは他のゲームで学んだ文法を持ち込む。
- **L-015** [rules] **勝敗条件はイベント駆動で即時評価**(状態が変わるたび)。「全滅後にまだ移動を求められる」のは、終了チェックがターン終端にしかないため。汎用: 条件達成の瞬間に残りの強制入力をすべてスキップする。
- **L-016** [design] **仕様は「輝く場面」を書いてから入れる**(オーナー指示)。制動は「ドリフト先に機雷」が無いと死に仕様だった。ハザード(岩・機雷)の導入は制動・押し込み・同士討ちを一気に輝かせた — 1つの地形が複数メカニクスの「場面」を供給する。essence監査#9に昇格。
- **L-017** [balance] **使われないリソース(被弾バーン0.1回/ラン)は難易度かコスパの警報**。使用率をシムの観測指標に含め、敵圧を上げる(or 効果を上げる)。難易度調整後バーン1.0回/ランに正常化。
- **L-019** [ux][qa] **「操作できない」報告の正体は大抵 ①開幕配置の経路封鎖 ②見えない否定(なぜ不可かの無表示) ③期待される操作文法の欠落**(ユニット先行選択)。コードのバグとは限らない — まず実機再現で「何が表示され何が無反応か」を観測する。対策の型: 配置の見直し+封鎖理由の可視化(⛔ツールチップ)+文法の受容(unit-first)。スポーン配置は一級バランス変数(L-008)なので変更後は必ずシム比較。
- **L-021** [ux] **強調の文法を1枚のルール表にして全画面に適用する**(SPEC §10.5): 一次=明滅cue(画面に常に1種類)/二次=静かな点線+👆説明/不可=⛔+理由/選択済み=固定枠。新しいUIを足すときは必ずこの4状態のどれかに割り当てる。「機能はあるが気づかれない」(ユニット先行選択)は二次状態の欠落が原因。
- **L-022** [visual] CSS疑似3Dの鉄則: ①盤はrotateX・ユニットは子要素でカウンター回転(ビルボード) ②`filter`/`opacity<1`/`overflow:hidden`は**preserve-3dを殺す**ので3D祖先に置かない(cueのfilterアニメで直立ユニットが潰れる事故を踏んだ) ③装飾は全てpointer-events:none ④perspectiveはコンテナへ。臨場感の投資順: 傾斜+直立>タイル厚み>接地影>視差・フォグ>ホバーリフト(調査出典はSPEC §10.6)。
- **L-020** [qa] **全コンテンツ×全アクション種のスモークテストを恒久化**(tests/sim.mjs)。「動かないものが多そう」への機械的回答 — カード追加のたびに自動で実行可否が検証される(42面PASS)。
- **L-025** [design] **複数の数値を持つ複合アクションは単値に畳む**(ラム: 突進力=移動距離=威力)。強化(+)が「どこに乗るか」が一意になり表示も予測も濁らない。
- **L-026** [design] StS研究より: 強化は「数値+1」より**機能変更**(デメリット除去=無反動化/対象変更=攻撃付与/キーワード付与=制動付与)が面白い。回復vs強化の焚き火天秤は「保険vs投資」という質の違う交換だから効く — **HPを通貨化**(船体持ち越し)して初めて成立する。カードプール比率は 汎用40:ビルド軸25:ピーキー35。
- **L-027** [design][juice] ボス演出のミニマム定石: WARNING帯→専用HPバー→パターンtelegraph→撃破の静寂→凱旋画面。報酬は「数え上げ・遅延・段階開示」で儀式化し、**周回用に演出ライト設定を必ず併設**(初見の感動は周回の苦痛)。
- **L-024** [tooling][art] **Codex CLI(gpt-image-2)でのアート自動生成パイプライン**: `'' | codex exec -s read-only --skip-git-repo-check -o <out> "<プロンプト>"`。落とし穴: ①stdinを閉じないと(`'' |`)無限待ち ②サンドボックスがワークスペースへの**バイナリ書込を拒否** → 生成物は `~/.codex/generated_images/` に落ちるので「最終メッセージにPNGの絶対パスだけ出せ」と指示してこちらでコピー(tools/genart.ps1) ③TaskStopはラッパーだけ殺してcodex子プロセスが残る — Stop-Processで掃除。プロンプト台帳は art/PROMPTS.md でスタイル統一。
- **L-023** [visual][process] **視覚QAは「自分の目で絵を見る」が最終工程**(tools/playrun.mjs=自動プレイ+ビート撮影)。シムと計算スタイル検証では拾えない欠陥が3ランで3つ出た: ①最高の瞬間(物理キル爆発)がモーダルに即隠される→**祝福を遮るUIは800ms待たせる** ②スコア/自己ベストが事務報告→**数字はデカく数え上げて祝う**(Balatro) ③同時スタンプの重なり潰れ→縦スタック。核心決断の画面(Leap or Keep)は「絵として」重さを持たせる(色対比+発光+大ボタン)。
- **L-018** [market] Steam売れ筋の中毒構造は優先度順に ①ビルド多様性 ②短ラン+自責設計 ③難易度ラダー ④**負けても得るドリップ** ⑤スコア。デイリー/FOMOは必須でない(Balatroの反例)。「敗北=無」になっていたら必ずドリップを足す。差別化フックは「1画面のGIFで伝わるか」で判定(本作=機雷誘爆チェーン)。出典: docs/steam-self-review.md
- **L-028** [design] **敵タイプは「プレイヤーの特定の行動をひとつ禁止/強制する」語彙として設計する**(アークナイツ敵設計の一般化): 重装甲=火力押しの禁止、支援=優先順位の強制、分裂=手数の圧迫、自爆=距離管理の強制、転移=キャンプの禁止。説明文に必ずカウンター(攻略法)を書く=敵図鑑が攻略本になる。実装面の教訓: 敵は DEFS+AI+desc の3点セットで「レジストリが複数」になるため、**出現テーブル全タイプの登録網羅をテストで機械検証する**(splitterのAI登録漏れ=遭遇した瞬間クラッシュ、をシムが深部到達時のみ検出した事故より)。
- **L-029** [design] **リスク自選システム(危機契約型)の鉄則**: ①タグは互いに直交させる(詰み合成の排除はバランス調整より設計で) ②実利報酬は線形・低めで打ち止め、高リスクには称号など名誉だけを置く(「実利は適度な挑戦で全回収、名誉は青天井」が荒れない) ③刺さる本質は「自分の苦手だけ避けて盛れる」自己表現性 — 同リスク値でも人ごとに構成が違うこと自体が価値。④新コンテンツ追加後のbot勝率低下は「難易度上昇」とは限らない — 本作ではサルベージ機会増→botがより深く潜って消耗死、という**行動変化**が主因だった。勝率だけでなく到達ゾーン・勝利時カーゴも見る。
- **L-030** [visual][ux] **preserve-3d文脈ではz-indexは死ぬ — クリック可能なオーバーレイは3Dごと持ち上げる**: 俯瞰モードで直立ビルボード化したユニットは、平置きのボタン(z-index:8)を奥行きで物理的に覆い、クリックを呑む(ラム方向矢印が押せないバグ)。対策: オーバーレイ自身も同じカウンター回転+translateZでユニットより手前に置く。L-022の系: 3D祖先の下に「2D の常識」(z-index/重なり順)を持ち込まない。検証は実マウスクリック(Playwright page.click=ヒットテスト)で行う — DOM存在チェックでは覆い隠しを検出できない。
- **L-031** [visual] **FLIPアニメはトーラスを知らない — ラップ移動は「退場→反対側から入場」の2相で描く**: 座標差分ベースのFLIPは縁越え移動を盤面横断の長スライドとして描き、直線移動と区別がつかない(オーナー報告)。端点比較だけでは縁越えと長距離直進を区別できないため、**ロジック側の全移動経路(ドリフト/移動/突進/押し引き/吸引/敵歩行)で縁越えステップを検出してイベント発行**し、UIが退場ゴースト+入場アニメに変換する。汎用: 空間が非ユークリッドなら「何が起きたか」は座標差分から復元できない — 移動の意味論はロジックが教えること。
- **L-032** [design][agent] **AIエージェント実プレイは最強のQA+記事素材**(実プレイ#001より): ①「表示ラベルと実数値の乖離」(強化+で移動1表示のまま移動2)のような人間が読み流すバグを、ラベルを根拠に行動するLLMは即座に踏んで報告する ②インターフェース設計の鉄則=合法手のID列挙(誤入力が構造的に不可能)+シード+選択ログのステートレスリプレイ(保存・再開・検証・改竄耐性がタダ) ③観測には「能力の理由」(敵desc)を含める — 人間はツールチップで補完するがエージェントは観測が全て ④同型選択肢の重複列挙は判断ノイズ(pair順序違い60→30に正規化)。根本教訓: **ラベルを手書きで持つな、specから生成しろ**(乖離はデータ駆動表示なら構造的に起きない — 今回は追従パッチで対処、次に乖離が出たら生成式に移行)。
- **L-033** [agent][design] **エージェントの「頂点宣言(wow)」は、機械のドラマ検出とは別レイヤーとして独立に記録する**: 航海記録を機械生成(物理キル等のイベント重み付け)するだけだと、プレイヤー/エージェントが本当に心を動かした瞬間(カードのサルベージ、約束の遂行など、ゲーム内イベントを伴わない行為)が完全に抜け落ちる。イナンナRun#003が「私の航海の背骨が記録から消えている」と指摘。対策: wowマークにコメントテキストを結びつけ、テキスト付きwowは独立した★行として本人の言葉で残す/テキストなしは近傍1件のみ★昇格(過剰昇格防止)。教訓の核: **自動要約は「観測可能な事実」しか拾えない。主観的な頂点は本人に宣言させ、別チャンネルで合流させる。** 併せて、時系列イベントを後からソートし直すと round=0 で記録される終端イベント等が紛れるので、挿入順(_seq)を時系列の正とし合成行だけを差し込む。
- **L-034** [agent][design] **エージェントの記憶は「想い出(情緒)」と「引き継ぎ(次回読む事実)」の2チャンネルに分ける**: 同じ記録を両用途に使うと、情緒の記憶は事実で薄まり、次回読む要約は感傷で重くなる。分離が正解。①想い出=chronicle(機械ドラマ)+voice(本人の声)+captain(自己像)を素材に、保存はそのまま散文/推奨は本人の声で蒸留。②引き継ぎ=軽量レコード(船/結果/型/note)を**エージェントが永続化し、new時に渡し返す**=ゲームはstatelessのまま任意のエージェントに汎用化できる。鍵は`note`(本人が書く心残り/抱負)で、これが次ランの冒頭に想起されると**ランをまたいだ物語の連続性**が生まれる(イナンナの「次は殲滅プロトコルの声を聞く」→次ラン回収が実証)。想起はゲーム冒頭1回だけ表示(毎ターンはノイズ)。
- **L-035** [visual][design] **連鎖演出の核心は「逐次性」— 同フレーム一括描画の連鎖はGIFで連鎖に見えない**: 再帰ロジックは一瞬で解決するが、演出は1段ずつ(本作110ms/段)ずらして波及を見せる。実装はfxイベントに遅延dを持たせ、ドレイン時にsetTimeoutで再キュー(ロジックは即時のまま=シム/エージェントに影響ゼロ)。エスカレーションは「同じ場所で育つ単一カウンタ」(誘爆×2→連鎖爆発×3→花火大会×N)が正解 — セル位置に段数スタンプを重ね置きすると異時刻・異セルの组合せで必ず文字が重畳する(L-023縦スタックでも防げない)。Codex処方で効いた順: 白芯(発光階層)>着弾ショックリング>連鎖ビーム>焦げ跡残光>HUD沈み。連鎖の「頻度」はfxでなく地形が作る — 機雷を6割で既存機雷の隣にクラスタ湧きさせたことでプレイヤーが狙って連鎖を組めるようになった(sonnetテスターが「1発で安定して連鎖する」と実証)。
- **L-036** [i18n][visual] **単一HTML+インライン和文のi18nは「DOM境界のMutationObserver+完全一致辞書+正規表現ルール」が最小侵襲**: ロジック/プロトコル/保存データは日本語のまま、#appへのaddedNodesだけを翻訳する(モーダル後差し込みもfxスタンプも自動で拾える)。ハマりどころ3つ: ①インライン要素(<b>等)でテキストノードは分割される — 辞書は「文」でなく「断片」単位で作る(実ページで未翻訳ノードを列挙するデバッグハーネスが必須、推測では潰れない) ②翻訳前にJSで加工される文字列(u.name.slice(0,6)等)は辞書に当たらない — 加工箇所で明示的にtrText() ③既存要素のtextContent更新はchildList変異でないのでObserverが拾わない — 更新系は明示trText()。ルールの捕獲群はEN_EXACT→再帰trTextで入れ子解決(旗艦ヴァグランツ→FS Vagrants)。辞書ソースはtools/i18n/*.part.json(ファイル名昇順マージ・先勝ち=aaa-fixが上書き層)、注入はtools/geni18n.mjs。
